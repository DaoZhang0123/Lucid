CHAPTER 01

Architecture

Pure-vision agent · vendor-neutral LLM · from feature to test

Lucid does not depend on MCP, application plug-ins, or app APIs. Any vision-capable LLM with tool-use can drive the loop — Lucid auto-discovers the models your subscription unlocks, so swapping providers is a config line, not a refactor. The four backends below are wired in today; adding a fifth is one Python file.

Anthropic API · claude-opus-4-5-20250929 GitHub Copilot OAuth · claude-opus-4.6 (and any model your subscription unlocks — /models auto-discovery) OpenAI · gpt-5.x (any OpenAI-compatible base URL) Google Gemini · gemini-2.x

Engine pipeline

Perception

Screenshot pyramid L0–L3 with a token budget. Default: don’t send pixels unless the model asks.

Decision

Vision-capable LLM with tool-use returns computer, launch_app, region, screenshot… The model is swappable; system prompt is the only place it’s pinned.

Safety

Danger-word filter, coordinate sanity check, HITL gate, PyAutoGUI fail-safe corner, panic hotkey.

Action

PyAutoGUI for clicks & scroll, Win32 SendInput for IME-safe typing, clipboard fallback for long text.

Verify

L3 pixel-diff around the cursor before and after each click; pixel_change_ratio < 0.5% warns “may have missed”.

Memory

Tips, icons, skills, templates merge into per-app tips.md and the icon atlas — what the agent learned (Chapter 4) is consumed here.

Every step writes a screenshot + tool JSON + verify diff into ~/.lucid/logs/<task>/events.jsonl. That trace is exactly what the test harness in Chapter 07 replays and judges — no fixtures, no mocks, the real desktop is the test bed.

Why pure-vision?

Zero integration cost

Works on private ERPs, RDP sessions, old Win32 apps, even games.

HITL safety

Three autonomy tiers: full, confirm_critical, confirm_each + panic hotkey + PyAutoGUI fail-safe.

Covers everything you can see

Slower than dedicated APIs, but no integration ever goes stale.

ReAct loop

observe
Capture screen at the level the model chose (L0–L3).
think
The LLM returns tool_use: computer, launch_app, region, …
safety
Safety Layer screens danger words, coordinates, HITL gate.
act
Execute via PyAutoGUI / Win32 SendInput / clipboard.
verify
Post-step L3 pixel-diff if click_verify_enabled.
repeat
Until finish() or budget hit.
CHAPTER 02

Screenshot pipeline

Four levels · token-budgeted · pixel-precise click verification

Images are expensive — don't send unless asked. The model is taught that "not seeing anything is normal; if you need to look, call screenshot()." The system attaches images only as a side-effect of a few high-value actions.

Four levels (L0–L3)

LevelScopeTriggerPurpose
L0Icon atlasscreenshot(level="icon_atlas")Identify unknown tray / taskbar icons
L1Full virtual desktopModel-initiatedCross-app situational awareness
L2Active window client areaSide-effect of launch_app / focus_window, onceForms, menus, list operations
L3~210×210 around the cursor, lime crosshairClick-verify, model-initiatedPixel-precise click confirmation, tooltip / small icon reading

Key choices

No auto post-step images

Removed in 2026-05. Saves 30–50% tokens on long threads.

L2 grid overlay

Only L2 carries a coordinate grid; system prompt teaches the model to read pixel coords from it.

L3 fixed tile + crosshair

No more UIA smart-sizing — predictable size keeps the model's coordinate mental model stable.

Click verify

L3 before/after every click — pixel_change_ratio < 0.5% warns "may have missed".

Recompression budget

Old screenshots are downscaled + JPEG'd, then replaced with a text placeholder. Forces "transcribe in the same turn".

DPI-aware coordinates

Per-Monitor V2, mss physical pixels, active-app pin defends against focus jumps.

Why this design?

Images are the single biggest token sink in a long thread, so we never auto-capture. Four explicit levels let the model ask for just enough pixels — L0 free, L1 cheap overview, L2 grid for reading coords, L3 the 210×210 tile that catches the most common failure (“clicked on nothing”). Re-compressing history and forcing in-turn transcription teaches the model a posture: “I don’t always need to look — I just need to act.” A 50-turn thread stays affordable where naive screenshotting would burn 80 % of the budget on pixels.

CHAPTER 03

Taskbar notification monitor

UIA snapshot sweep · visual diff fallback · per-app preference

Action Center (WinRT UserNotificationListener) misses Teams / WeChat too often, so Lucid runs two parallel channels. UIA fires first when the app cooperates; the visual diff is the safety net for everything else.

Channel A — UIA snapshot sweep LIVE

Button-class whitelist

Only Taskbar.TaskListButtonAutomationPeer (Win11 app buttons) and SystemTray.NormalButton (tray icons). Clock, volume, network, Show-Desktop sliver and chevron are filtered at the handler boundary — no log floods.

500 ms snapshot sweep

Win11 22H2+ XAML taskbar buttons accept PropertyChanged handlers but never call them back — UIA events alone miss everything. Lucid walks the dozen buttons every 500 ms, reads two properties, edge-detects against the previous snapshot. <1 % of one core; tighter doesn’t help, looser misses WeChat’s ~1 s flash.

Localised unread hints

Name matched against 未读 / 新消息 / 新活动 / unread / new message / new activity; HelpText matched against 已请求注意 / attention requested / flashing. WeChat never changes Name — only the HelpText flash — so we cover both.

lucid/taskbar_uia_monitor.py · raw signals at logs/taskbar-monitor/uia-events.jsonl · config [taskbar_uia]

Channel B — Visual two-step (fallback) LIVE

Crop
Auto-detect the real taskbar height from FindWindow("Shell_TrayWnd"); default skip 100 px on the left (Win11 weather widget) and 100 px on the right (clock) so they never enter the strip.
Diff
Every poll_interval_sec (2 s default), dHash or pixel-diff over the strip; X-axis projection segments the changed columns so each segment is scored on its own.
LLM confirm
If a segment exceeds the threshold, attach the cropped strip and ask “Is this a new chat notification? Which app?” — structured output. Rejected segments enter a 30 s cooldown (±20 px) so a hover / focus underline on the same icon doesn’t burn LLM calls.

lucid/taskbar_monitor.py · config [visual_notify]

Arbitration & per-app preference

  • UIA hit → calls bump_external_cooldown() on the visual monitor, suppressing Step-2 LLM for visual_suppress_after_uia_sec (20 s default).
  • ~/.lucid/taskbar_sources.json stores per-app source preference: ["uia"] only, ["visual"] only, or ["uia", "visual"] with a primary. Defaults: Teams / Outlook / Slack / WeChat → UIA only. Visual stays as the safety net for the long-tail apps that paint their own badges without exposing UIA hints.
  • Doze drains logs/taskbar-monitor/learn-queue.jsonl — the queue carries both confirmed and rejected events, so the LLM also learns “this widget is not a notification” from counter-examples (Chapter 4).
  • Confirmed events are deduped through the strict-mode queue and can fire auto_chat_instruction to spawn an auto-reply task.

Why this design?

We tried each channel alone first. UIA alone is fast and free but silent on apps that paint their own badges (WeChat). Visual diff alone burns LLM calls on every clock tick and weather widget. The lesson: notifications are heterogeneous — no single mechanism covers them all. Two parallel channels plus a per-app preference file let the system learn who to trust on which path: most apps end up UIA-only (Teams, Outlook, Slack, WeChat after the HelpText flash route was found); visual stays as the safety net for the long tail. Counter-examples flow back through doze, and the steady-state cost trends toward zero.

CHAPTER 04

Doze learning

Low-priority reflection on idle · learns tips and taskbar widgets

When does it wake?

Every tick_interval_sec (60 s default) the worker checks four conditions; all must hold:

No running worker

Doze never overlaps itself or task threads.

Task queue empty

Nothing else waiting.

Idle ≥ 300 s

now − last_user_activity_ms ≥ idle_threshold_sec.

Not already dozing

Re-entry guarded.

User activity means state-changing RPCs (start_task, thread_new, …). Polling like ping / get_status doesn't count. Any user activity during a pass set()s the cancel flag — the user never waits for doze.

Text channel

One pass reads one unprocessed thread, builds a compact transcript (no images), shows the LLM existing tips / memory digest, and exposes only learn_tip / remember / load_app_tips. Output goes to tools.md and apps/<slug>/tips.md. Logs split per day at ~/.lucid/doze-YYYYMMDD.log.

Taskbar-visual learning

Visual Step 2 writes "diff seen but classifier said no" counter-examples to logs/taskbar-monitor/learn-queue.jsonl. Doze drains the queue, groups by (app_slug, signature), and after ≥ 3 examples writes a [taskbar_visual] tip into apps/<slug>/tips.md. Next Step 2 call sees the tip via load_app_tips and stops re-flagging the same widget.

Safety: the doze worker can never call GUI tools — the dispatch whitelist refuses anything outside its small tool set. Doze reads events.jsonl (no base64 images), not messages.json. Off by default; enable in /settings → Doze.

Why this design?

Most agents only learn when the user gives explicit feedback. But the richest signal is “what just happened, viewed in hindsight.” Idle time is free — CPU is there, the user is away, the trace already exists. Doze is a low-priority secondary worker dispatching to a small write-only tool set (learn_tip, remember); it can never click anything, and is preempted the moment the user moves. The result is quiet, continuous self-improvement that never makes you wait.

CHAPTER 05

Skills

Anthropic Agent Skills format · drop a folder, get a capability

A skill is a folder containing a SKILL.md with YAML frontmatter and a Markdown body — the exact format used by Anthropic's official Agent Skills. Drop a skill folder into ~/.lucid/skills/<slug>/ and Lucid picks it up.

Format

---
name: weekly-report
description: Open Outlook and draft a weekly report email.
version: "1.0"
license: MIT
source: user
---

# Weekly report

1. launch_app: outlook
2. ...

Skills vs Templates

TemplatesSkills
ShapePlain instruction stringSKILL.md (YAML + body, can ship scripts)
TriggerUser clicks RunUser clicks Use or agent decides from system prompt
System promptNot injectedOne line per skill: name (tag): description

Progressive disclosure

list_skills()

Enumerate available skills.

read_skill(slug)

Pull the full SKILL.md body.

read_file(path)

Open reference/*.md, scripts/*.py, any companion file.

Online install (LLM-led)

The README's opening example demonstrates the recommended bootstrap:

run_shell("git clone https://github.com/anthropics/skills ~/.lucid/skills")
read_file("~/.lucid/skills/pptx/SKILL.md")
learn_tip("PPTX skill installed at ~/.lucid/skills/pptx; entry: build.py")
run_shell("python ~/.lucid/skills/pptx/build.py proposal.docx proposal.pptx")

RPC install_skill_url is restricted to user action (not LLM-triggered) and tags the result with source: "online" (untrusted) so the safety layer can decide.

Why this design?

There are already excellent skills in the wild — Anthropic's official catalogue, community repos, internal corporate knowledge bases. Reinventing them per user is wasted effort and dilutes quality. We adopted Anthropic's open SKILL.md format on purpose: any skill anyone has written elsewhere works here as-is, and the main loop becomes discovery — finding and adopting a good skill — not authoring. Progressive disclosure (one-line catalog → read_skill on demand) means a hundred skills cost almost nothing in tokens until the agent actually needs one.

CHAPTER 06

Voice input

Push-to-talk · fully offline · focus-aware

Speak a sentence — or just type one — and Lucid either opens a new thread, aborts the current one, or appends to the input box. Voice and typed text share the same intent classifier (see below) so “cancel that” works whether you say it or type it.

Trigger

Hold Space ≥ 5 s

Accessibility-first: no modifier combos by default. A WH_KEYBOARD_LL hook handles it globally; a webview-level fallback covers the case where Lucid’s own window has focus (Chromium otherwise eats the keystroke).

Configurable

hotkey, hold_threshold_ms, stop_mode (release / tap_again / auto_silence).

Focus-aware

When a text input or IME composition is active, hold is ignored — Space stays Space.

ASR — local, offline

faster-whisper

Other engines (sherpa-onnx / vosk) kept for back-compat but hidden from UI.

Local weights

No cloud round-trip; everything stays on the box.

≤ 1.5 s

Stop → text on CPU for the small models.

Intent dispatch LIVE

After transcription — or, for short typed messages (≤ 100 chars) sent while a thread is running — a small classification LLM (~150 max tokens) decides one of three actions:

thread_new
Clearly a new task ("open Notepad and write hello").
thread_abort
Stop the current run ("cancel", "stop", "never mind"). Routed through voice_dispatch_abort as a priority-0 urgent thread; the prompt is source-aware (says typed or said (voice)) so the LLM never invents a phantom voice request.
dictation_append
Extend the input box ("…and also mention…").

If confidence is low, the voice overlay shows quick-pick chips when it falls back to result mode (no main-window dictation sink). The overlay sits top-centre with waveform + countdown.

Why this design?

Two reasons voice was built in from day one:

  1. Speed for everyone. Speaking is roughly 3× faster than typing for most people, and you can do it while reading the screen. Most desktop assistants assume keyboard-only input — an unnecessary ceiling.
  2. Accessibility. The keyboard isn't an option for users with motor disabilities, RSI, or a temporary injury. A long-press hotkey + fully offline ASR means the entry point works without trusting the cloud or installing a separate accessibility stack.

Push-to-talk (instead of an always-on wake word) keeps the CPU cold, never leaks audio, and avoids the “wake-word fatigue” that makes phone assistants feel unreliable.

CHAPTER 07

End-to-end test loop

One JSON · one Python runner · one SKILL.md · Copilot drives the iteration

The whole regression harness for a GUI agent is built from three files: a queries.json with ~60 real-desktop tasks, a 200-line run.py that only enqueues them into the sidecar’s priority queue, and a SKILL.md that teaches VS Code Copilot how to read the resulting manifest.json and write the next iteration’s fix plan. No pytest, no fixtures, no per-app mocks — the desktop itself is the test rig.

Inputs — three plain files

queries.json

~60 real tasks across 15 categories (Notepad, Calculator, Edge, Office, WeChat, Teams…), each with an instruction and optional expect_files. Adding coverage is one JSON object.

run.py — an enqueuer

Boots the sidecar, fires start_task N times 300 ms apart, watches each thread’s events.jsonl mtime, and cancels just that task if it stalls past --per-task-cap s. The sidecar’s own priority queue does the scheduling.

SKILL.md

Anthropic Agent Skill that teaches VS Code Copilot the analysis protocol: verdict rules, subagent fan-out, severity / category-of-fix contract, output language. The analysis is a Copilot prompt — not Python.

Iteration loop EVERY RELEASE

Run
python Test\run.py — the sidecar executes ~60 queries on the real desktop (don’t pre-clean it; clutter is part of the test). Each query backfills status, final_text and a deliverables_snapshot (file exists / size / mtime / contains-match, pinned at task close) into manifest.json.
Judge
Three-layer verdict per query: run (status == "ok") × goal (a per-thread Copilot subagent reads instruction + final_text + context.log + 2–3 step screenshots and returns pass / partial / fail) × deliverables (expect_files snapshot). No brittle expect_signal grep — paraphrases pass, hallucinated success doesn’t.
Iterate
Subagents fan out (4–6 in parallel); each ends with a machine-readable severity: / category_of_fix: pair (prompt / tool / config / new-region / new-app-spec / bug-fix). Copilot clusters them into iteration-plan.md: high-leverage fixes (≥ 2 agree) first, single-thread blockers second, full per-thread table last. Pass a baseline run to surface regressions.

Net effect: every code change lands with a one-paragraph evidence trail (which categories regressed, which fix unblocked them) instead of vibes; onboarding a new app is “add one JSON entry, re-run, read the plan.”

Why this design?

Classic GUI test frameworks (Playwright, pytest+Selenium) make the developer spell out every assertion in code. But assertion is exactly what the agent should do — it knows what the goal looks like. So we flipped it: queries.json is the only contract, Copilot is the judge. Three benefits: (a) adding a case is one JSON line, not a fixture; (b) the judge tolerates paraphrase (“a weekly report email” ↔ “draft email”), so brittle string matches are gone; (c) the same SKILL.md that judges this release also writes the iteration plan for the next one. Tests and the learning loop become the same loop.

CHAPTER 02

截图流水线

四个等级 · token 预算可控 · 像素级点击核验

图像很贵——不点名不送。系统教模型："看不到东西很正常；要看就主动调 screenshot()。"只有少数高价值动作会顺带挂图。

四个等级（L0–L3）

等级范围触发用途
L0图标库screenshot(level="icon_atlas")识别未知托盘 / 任务栏图标
L1整个虚拟桌面模型主动跨应用态势感知
L2活动窗口客户区launch_app / focus_window 顺带一次表单、菜单、列表操作
L3光标周围 ~210×210，绿色十字点击核验、模型主动像素级点击确认、tooltip / 小图标

关键取舍

不再事后自动挂图

2026-05 移除。长会话省 30–50% token。

L2 坐标网格

只在 L2 叠网格；系统提示教模型直接读像素坐标。

L3 定尺寸 + 十字

取消 UIA 自适应；尺寸稳定，模型的坐标心智才稳定。

点击核验

每次点击前后各取一张 L3；pixel_change_ratio < 0.5% 即提示"可能点空"。

重压缩预算

历史截图缩放 + JPEG 重压，再替换成文字占位。强制"同回合内转写关键信息"。

DPI 感知坐标

Per-Monitor V2，mss 物理像素，活动应用钉死，防焦点漂移。

为什么这么设计

截图是长会话里最大头的 token 成本，所以从不自动截。四个层级让模型按需点菜——L0 免费、L1 便宜的全屏、L2 带网格用于读坐标、L3 那张 210×210 的小图专治"以为点中其实点空"。历史截图重压 + 同回合内转写关键信息，是在教模型一种姿态："很多时候不需要再看，只需要做"。50 回合的长任务也能扛得住成本——如果每步都截图，光像素就要烧掉 80% 的预算。

CHAPTER 03

任务栏通知监听

UIA 快照轮询 · 视觉 diff 兜底 · 按应用偏好

Action Center（WinRT UserNotificationListener）在 Teams / 微信上不可靠，因此 Lucid 启了两条并行通道。应用配合时 UIA 先到，其他情况由视觉 diff 兜底。

通道 A — UIA 快照轮询 已落地

按钮类名白名单

只看 Taskbar.TaskListButtonAutomationPeer（Win11 应用按钮）和 SystemTray.NormalButton（托盘图标）。时钟、音量、网络、Show Desktop 缝隙、Chevron 在 handler 边界直接过滤——日志不会被冲爆。

500 ms 快照轮询

Win11 22H2+ XAML 任务栏按钮接受 PropertyChanged handler 但永远不回调——光靠事件流什么都收不到。所以 Lucid 每 500 ms 扫一遍那十几个按钮，读两个属性，跟上一张快照做边沿检测。单核 <1%；更短意义不大，更长会漏微信约 1 秒的 flash。

多语言未读提示

Name 对照 未读 / 新消息 / 新活动 / unread / new message / new activity；HelpText 对照 已请求注意 / attention requested / flashing。微信不改 Name——只改 HelpText 的 flash 标志——这套规则两边都覆盖。

lucid/taskbar_uia_monitor.py · 原始信号落到 logs/taskbar-monitor/uia-events.jsonl · 配置位于 [taskbar_uia]

通道 B — 视觉两步（兜底） 已落地

裁剪
用 FindWindow("Shell_TrayWnd") 自动检测真实任务栏高度；默认左侧跳过 100 px（Win11 天气小部件）、右侧跳过 100 px（时钟），它们永远不会进入 strip。
Diff
每 poll_interval_sec（默认 2 秒）对 strip 做 dHash 或像素 diff；X 方向投影把变化列切成段，逐段评分。
LLM 确认
某段超过阈值时，附上裁剪片，问“是不是新消息？哪个应用？”，输出结构化。被拒分段进 30 秒冷却（±20 px）——同一图标的 hover / 焦点下划线不会反复烧 LLM。

lucid/taskbar_monitor.py · 配置位于 [visual_notify]

仲裁 & 按应用偏好

  • UIA 命中后调用视觉通道的 bump_external_cooldown()，第二步 LLM 被压制 visual_suppress_after_uia_sec（默认 20 秒）。
  • ~/.lucid/taskbar_sources.json 存每个应用的通道偏好：仅 ["uia"]、仅 ["visual"]、或 ["uia", "visual"] 双开并指定 primary。默认：Teams / Outlook / Slack / 微信都走仅 UIA。视觉只给那些自己画 badge 、又不暴露 UIA 提示的长尾应用兼底。
  • 打盹排干 logs/taskbar-monitor/learn-queue.jsonl——队列里同时包含确认事件和被拒事件，LLM 也会从反例里学到“这个小部件不算通知”（见第 4 章）。
  • 确认事件经 strict 模式队列去重，可触发 auto_chat_instruction 自动回复任务。

为什么这么设计

我们先把两条通道分别试过。单 UIA 又快又便宜，但碰上自己画 badge 的应用（微信）就哑了；单视觉 diff 又每秒时钟、每个天气小部件都要烧 LLM。教训是：通知本来就是异构的，单一机制覆盖不了。两条并行通道加一份按应用偏好的文件，让系统去学——大多数应用最后都落到仅 UIA（Teams、Outlook、Slack，以及找到 HelpText flash 路径后的微信），视觉只给长尾应用兼底。被拒事件经打盹回流，稳态成本会越来越接近零。

CHAPTER 04

打盹学习

空闲时低优先反思 · 学 tip 和任务栏小部件

什么时候会醒？

每 tick_interval_sec（默认 60 秒）worker 检查四个条件，必须全部满足：

没在跑 worker

打盹绝不与任务并发。

任务队列空

没有别的活儿在等。

空闲 ≥ 300 秒

now − last_user_activity_ms ≥ idle_threshold_sec。

当前未在打盹

再入保护。

用户活动指改变状态的 RPC（start_task、thread_new……）；轮询类如 ping、get_status 不算。打盹期间出现用户活动会立即 set() 取消标志——用户永远不会等打盹。

文本通道

一次只读一条未处理线程，构造紧凑文本（无图），把现有 tip / 记忆摘要也喂给 LLM，仅暴露 learn_tip / remember / load_app_tips。落地到 tools.md 与 apps/<slug>/tips.md。日志按天切到 ~/.lucid/doze-YYYYMMDD.log。

任务栏视觉学习

视觉第二步若"看到 diff 但分类器说不算"，会把反例写到 logs/taskbar-monitor/learn-queue.jsonl。打盹排干该队列，按 (app_slug, signature) 聚合，达阈值（默认 ≥ 3）后向 apps/<slug>/tips.md 写入 [taskbar_visual] tip。下一次第二步通过 load_app_tips 看到 tip，就不再误报同一小部件。

安全：打盹 worker 永远不能调 GUI 工具——dispatch 白名单只放它那一小撮。打盹只读 events.jsonl（已序列化，无 base64 图），不读可能含密的 messages.json。默认关闭，在 /settings → 打盹 启用。

为什么这么设计

多数智能体只在用户给反馈时才学。但最丰厚的学习信号其实是"刚才发生的事，事后冷静看一遍"。空闲时间是免费的——CPU 在那、用户不在、trace 已经写好了。打盹是低优先的二级 worker，dispatch 白名单只放写类工具（learn_tip、remember），永远点不了 GUI，用户一动手就立刻让位。换来的是安静、连续、不打扰的自我改进。

CHAPTER 06

语音输入

按住说话 · 完全离线 · 焦点感知

说一句话——或者直接打字——Lucid 自动决定是开新线程、打断当前线程、还是追加到输入框。语音转写和键入文本走同一个意图分类器，所以无论你说还是打字，“取消” 都管用。

触发

长按 Space ≥ 5 秒

无障碍优先：默认不要组合键。全局走 WH_KEYBOARD_LL，Lucid 自身窗口聚焦时还有一个 webview 级 fallback（否则 Chromium 会把 Space 吃掉）。

可配置

hotkey、hold_threshold_ms、stop_mode（release / tap_again / auto_silence）。

焦点感知

文本框或输入法合成中时，长按被忽略——Space 还是 Space。

ASR——本地离线

faster-whisper

其他引擎（sherpa-onnx / vosk）保留向后兼容但不进 UI。

本地模型权重

不上云，全部留本机。

≤ 1.5 秒

CPU 上 small 模型停录 → 文字。

意图分发 已落地

转写之后——或者，运行中输入框打入的短文本（≤ 100 字）——一个小分类 LLM（独立短系统提示，~150 token）从三种动作里挑一：

thread_new
显然是新任务（"打开记事本写 hello"）。
thread_abort
叫停当前（"取消"、"停下"、"算了"）。走 voice_dispatch_abort，作为 priority=0 紧急线程入队；prompt 区分来源（typed 或 said (voice)），不会把打字的请求伪装成语音。
dictation_append
补充输入框（"……再加一句……"）。

仅当浮层退到 result 兜底（没有主窗 sink）时才弹快选 chips。浮层挂在屏幕顶部居中，带波形和倒计时。

为什么这么设计

一开始就把语音做进去，原因有两个：

  1. 对每个人都更快。绝大多数人说话比打字快 3 倍左右，而且能一边看屏幕一边说。多数桌面助手默认只接受键盘输入——这个天花板其实没必要存在。
  2. 无障碍。对运动障碍、RSI 劳损、临时受伤、或者暂时不方便打字的用户来说，键盘根本不是选项。长按热键 + 完全本地的 ASR 意味着这个入口既不用信云、也不用单独装无障碍栈就能用。

按住说话（而不是常听唤醒词）让 CPU 一直凉、永不漏音频，也不会有手机助手那种"唤醒词总不灵"的疲劳感。

CHAPTER 07

端到端回归测试

一个 JSON · 一个 Python runner · 一份 SKILL.md · Copilot 负责分析与迭代

整套 GUI agent 的回归测试架子就三个文件：queries.json 里 60 来个真桌面任务、200 行的 run.py 只负责把任务推进 sidecar 的优先队列、再加一份 SKILL.md 告诉 VS Code Copilot 怎么读跑出来的 manifest.json、怎么写下一轮的修复计划。没有 pytest、没有 fixture、没有 per-app mock — 真桌面本身就是夹具。

输入：三个普通文件

queries.json

15 个类别下 60 来个真任务（记事本、计算器、Edge、Office、微信、Teams…），每条带 instruction 和可选的 expect_files。扩 case 就是加一段 JSON。

run.py — 只入队

启 sidecar，每 300 ms 喂一条 start_task；盯各线程 events.jsonl 的 mtime，超过 --per-task-cap s 没动就只 cancel 这一条，其余继续跑。调度交给 sidecar 自己的优先队列。

SKILL.md

Anthropic Agent Skill 标准格式，把“怎么分析这一轮跑”写成给 VS Code Copilot 的协议：判定规则、子代理 fan-out、severity / category_of_fix 合约、输出语言。分析逻辑是 prompt，不是 Python。

迭代循环 每个 RELEASE

跑
python Test\run.py — sidecar 在真桌面跑 60 来条 query（别预清理，桌面乱本来就是测试条件）。每条收尾时把 status、final_text 和 deliverables_snapshot（文件是否存在 / 大小 / mtime / 子串命中，定格在 task close 那一刻）回填进 manifest.json。
判
每条 query 三层判定：运行层（status == "ok"）× 目标层（per-thread Copilot 子代理读 instruction + final_text + context.log + 2–3 张 step 截图，返 pass / partial / fail）× 交付层（expect_files 快照）。不再做脆弱的 expect_signal 子串匹配 — 同义说法照样过、模型谎称完成照样挂。
迭
子代理 fan-out（4–6 个并行），每个结尾必带机器可读的 severity: / category_of_fix:（prompt / tool / config / new-region / new-app-spec / bug-fix）。Copilot 把它们聚类成 iteration-plan.md：≥2 个子代理同意的高杠杆改动排第一、单线程 blocker 第二、完整逐线程表第三。传一个 baseline 还能跟上一轮对比看回归。

效果：每次改代码都带一段“哪些类别回归了 / 哪个修复解锁了它们”的证据，不是凭感觉；新接一个 app 就是“加一条 JSON、重跑、读 plan”。

为什么这么设计

经典 GUI 测试框架（Playwright、pytest+Selenium）要求开发者把每条断言都写成代码。但"断言"这件事本来就该交给智能体——它才知道目标长什么样。所以我们把它翻过来：queries.json 是唯一的契约，Copilot 是裁判。这样有三个好处：(a) 加一条 case 就是加一行 JSON，不用写 fixture；(b) 裁判容忍同义表达（"一封周报邮件" ↔ "邮件草稿"），脆弱的字符串匹配消失了；(c) 评判这一版的同一份 SKILL.md，会顺手产出下一版的迭代计划。测试和学习闭环合并成了同一条回路。

CHAPITRE 01

Architecture

Agent purement visuel · LLM agnostique · de la fonctionnalité au test

Lucid ne dépend ni de MCP, ni de plug-ins, ni d'API d'application. N'importe quel LLM multimodal avec tool-use peut piloter la boucle — Lucid détecte automatiquement les modèles que votre abonnement débloque ; changer de fournisseur est une ligne de config, pas un refactor. Les quatre backends ci-dessous sont câblés aujourd'hui ; en ajouter un cinquième tient en un fichier Python.

API Anthropic · claude-opus-4-5-20250929 GitHub Copilot OAuth · claude-opus-4-6 (et tout modèle débloqué par l'abonnement) OpenAI · gpt-5.x (toute base URL compatible OpenAI) Google Gemini · gemini-2.x

Pipeline du moteur

Perception

Pyramide de captures L0–L3 avec budget de tokens. Par défaut : aucun pixel envoyé tant que le modèle ne demande pas.

Décision

LLM multimodal avec tool-use renvoie computer, launch_app, region, screenshot… Le modèle est interchangeable ; il n'est nommé que dans le system prompt.

Sécurité

Filtre de mots dangereux, plausibilité des coords, HITL, coin fail-safe PyAutoGUI, raccourci panique.

Action

PyAutoGUI pour clics & scroll, Win32 SendInput pour saisie IME-safe, presse-papier en repli sur les longs textes.

Vérification

Diff pixel L3 autour du curseur avant et après chaque clic ; pixel_change_ratio < 0,5 % avertit « clic peut-être manqué ».

Mémoire

Astuces, icônes, skills, templates fusionnent dans tips.md par app et dans l'atlas d'icônes — ce qu'apprend l'agent (chapitre 4) est consommé ici.

Chaque étape écrit une capture + le JSON de l'outil + le diff de vérification dans ~/.lucid/logs/<tâche>/events.jsonl. Le harnais de tests du chapitre 7 rejoue et juge exactement cette trace — pas de fixtures, pas de mocks, le vrai bureau est le banc d'essai.

Pourquoi le tout-visuel ?

Zéro coût d'intégration

Marche sur ERP privé, RDP, vieilles apps Win32, voire jeux.

Sécurité HITL

Trois paliers : full, confirm_critical, confirm_each + raccourci panique + fail-safe PyAutoGUI.

Couvre tout ce qui se voit

Plus lent qu'une API dédiée, mais aucune intégration ne casse.

CHAPITRE 02

Pipeline de capture

Quatre niveaux · budget token maîtrisé · vérification de clic au pixel

Les images coûtent cher — n'envoyez que sur demande. Le système enseigne au modèle : « ne rien voir est normal ; si tu as besoin de regarder, appelle screenshot(). » Quelques actions à forte valeur en attachent une au passage.

Quatre niveaux (L0–L3)

NiveauPortéeDéclencheurUsage
L0Atlas d'icônesscreenshot(level="icon_atlas")Identifier une icône tray / barre inconnue
L1Bureau virtuel completÀ l'initiative du modèleConscience inter-applications
L2Zone cliente de la fenêtre activeEffet de bord de launch_app / focus_window, une foisFormulaires, menus, listes
L3~210×210 autour du curseur, croix verteVérification de clic, ou à l'initiative du modèleConfirmation au pixel, lecture tooltip / petite icône

Choix clés

Plus d'image auto post-étape

Retiré en 2026-05. Économise 30–50 % de tokens sur les longs fils.

Grille de coordonnées L2

La grille n'apparaît qu'en L2 ; le prompt enseigne à lire les coords pixel directement.

L3 taille fixe + croix

Plus de dimensionnement UIA : taille stable → représentation des coords stable.

Vérification du clic

L3 avant/après chaque clic ; pixel_change_ratio < 0,5 % → alerte « clic peut-être manqué ».

Budget de recompression

Les vieilles captures sont réduites + JPEG, puis remplacées par un placeholder texte — force la transcription dans le même tour.

Coords DPI-aware

Per-Monitor V2, pixels physiques via mss, app active épinglée contre les sauts de focus.

Pourquoi ce design ?

Les images sont le poste de tokens le plus lourd d'un long fil — donc on ne capture jamais automatiquement. Quatre niveaux explicites laissent le modèle commander juste ce qu'il faut : L0 gratuit, L1 vue d'ensemble bon marché, L2 grille pour lire les coords, L3 la tuile 210×210 qui attrape le mode d'échec le plus fréquent (« cliqué sur rien »). Recompresser l'historique et forcer la transcription au même tour enseigne au modèle une posture : « souvent pas besoin de regarder, juste agir ». Un fil de 50 tours reste abordable là où une capture naïve à chaque pas brûlerait 80 % du budget en pixels.

CHAPITRE 03

Notifications de la barre des tâches

Sweep UIA · diff visuel en repli · préférence par app

Le Centre d'action (WinRT UserNotificationListener) rate régulièrement Teams / WeChat : Lucid fait tourner deux canaux en parallèle. UIA parle en premier quand l'app coopère ; le diff visuel est le filet de sécurité pour le reste.

Canal A — Sweep UIA EN PROD

Liste blanche de classes

Uniquement Taskbar.TaskListButtonAutomationPeer (boutons d'app Win11) et SystemTray.NormalButton (icônes tray). Horloge, volume, réseau, fente Show-Desktop et chevron sont filtrés à la frontière du handler — pas d'inondation de logs.

Sweep 500 ms

Sur Win11 22H2+ la barre XAML accepte les handlers PropertyChanged mais ne les rappelle jamais — les événements seuls ne voient rien. Donc Lucid balaie la douzaine de boutons toutes les 500 ms, lit deux propriétés, détection d'edge contre le snapshot précédent. <1 % d'un cœur ; plus court n'aide pas, plus long rate le flash WeChat d'~1 s.

Indices d'unread localisés

Name comparé à 未读 / 新消息 / 新活动 / unread / new message / new activity ; HelpText comparé à 已请求注意 / attention requested / flashing. WeChat ne change pas le Name — uniquement le flash HelpText — donc on couvre les deux.

lucid/taskbar_uia_monitor.py · signaux bruts dans logs/taskbar-monitor/uia-events.jsonl · configuration [taskbar_uia]

Canal B — Visuel deux étapes (repli) EN PROD

Découper
Hauteur réelle de la barre auto-détectée via FindWindow("Shell_TrayWnd") ; par défaut 100 px ignorés à gauche (météo Win11) et 100 px à droite (horloge), donc ils n'entrent jamais dans la bande.
Diff
Toutes les poll_interval_sec (2 s par défaut), dHash ou diff pixel sur la bande ; la projection sur X découpe les colonnes qui ont bougé et chaque segment est noté à part.
Confirmation LLM
Un segment au-dessus du seuil → on attache la bande et on demande « nouvelle notif ? quelle app ? » — sortie structurée. Les segments rejetés entrent dans un cooldown de 30 s (±20 px) pour qu'un survol ou un soulignement de focus sur la même icône ne brûle pas de LLM.

lucid/taskbar_monitor.py · configuration [visual_notify]

Arbitrage & préférence par app

  • Sur un hit UIA, appel à bump_external_cooldown() sur le moniteur visuel : l'étape 2 du LLM est inhibée pendant visual_suppress_after_uia_sec (20 s par défaut).
  • ~/.lucid/taskbar_sources.json stocke la préférence par app : ["uia"] seul, ["visual"] seul, ou ["uia", "visual"] en double avec un primary. Par défaut : Teams / Outlook / Slack / WeChat en UIA seul. Le visuel reste le filet de sécurité pour les apps de longue traîne qui dessinent leur propre badge sans exposer d'indice UIA.
  • Doze vide logs/taskbar-monitor/learn-queue.jsonl — la file contient à la fois les événements confirmés et rejetés, donc le LLM apprend aussi « ce widget n'est pas une notif » à partir des contre-exemples (chapitre 4).
  • Les événements confirmés sont dédupliqués en mode strict et peuvent déclencher une auto-réponse via auto_chat_instruction.

Pourquoi ce design ?

On a essayé chaque canal seul d'abord. UIA seul est rapide et gratuit mais muet sur les apps qui dessinent leur propre badge (WeChat). Le diff visuel seul brûle un appel LLM à chaque tic d'horloge ou widget météo. Leçon : les notifications sont hétérogènes — aucune mécanique unique ne couvre tout. Deux canaux parallèles + un fichier de préférence par app : le système apprend qui croire sur quel chemin — la plupart des apps finissent en UIA seul (Teams, Outlook, Slack, et WeChat depuis qu'on a trouvé la route HelpText flash) ; le visuel reste le filet de sécurité pour la longue traîne. Les contre-exemples reviennent par Doze, et le coût en régime stable tend vers zéro.

CHAPITRE 04

Apprentissage Doze

Réflexion basse priorité au repos · apprend astuces et widgets

Quand se réveille-t-il ?

Toutes les tick_interval_sec (60 s par défaut), le worker vérifie quatre conditions ; toutes doivent tenir :

Aucun worker en cours

Doze ne se chevauche jamais avec une tâche.

File de tâches vide

Rien d'autre n'attend.

Inactif ≥ 300 s

now − last_user_activity_ms ≥ idle_threshold_sec.

Pas déjà en doze

Protection ré-entrée.

L'activité utilisateur désigne les RPC qui changent l'état (start_task, thread_new…). Les sondes (ping, get_status) ne comptent pas. Toute activité pendant une passe positionne le flag d'annulation — l'utilisateur n'attend jamais Doze.

Canal texte

Une passe lit un fil non traité, monte une transcription compacte (sans image), expose au LLM les tips et le digest mémoire, et n'autorise que learn_tip / remember / load_app_tips. Écrit dans tools.md et apps/<slug>/tips.md. Journaux quotidiens dans ~/.lucid/doze-YYYYMMDD.log.

Apprentissage visuel de la barre

L'étape 2 visuelle écrit ses contre-exemples (« diff vu, classifieur a dit non ») dans logs/taskbar-monitor/learn-queue.jsonl. Doze draine la queue, groupe par (app_slug, signature), et au-delà du seuil (≥ 3) écrit un tip [taskbar_visual] dans apps/<slug>/tips.md. L'appel suivant voit le tip via load_app_tips et arrête de flagger le widget.

Sécurité : le worker Doze ne peut jamais appeler d'outil GUI — la whitelist refuse tout en dehors de son petit set. Doze lit events.jsonl (pas d'image base64), pas messages.json. Désactivé par défaut ; à activer dans /settings → Doze.

Pourquoi ce design ?

La plupart des agents n'apprennent que sur retour explicite. Mais le signal le plus riche, c'est « ce qui vient de se passer, revu à tête reposée ». Le temps d'inactivité est gratuit — le CPU est là, l'utilisateur absent, la trace déjà écrite. Doze est un worker secondaire à priorité stricte, limité à un petit set d'outils en écriture (learn_tip, remember) ; il ne peut jamais cliquer, et il est préempté dès que l'utilisateur bouge. Résultat : auto-amélioration silencieuse et continue, sans jamais vous faire attendre.

CHAPITRE 05

Skills

Format Anthropic Agent Skills · un dossier, une nouvelle capacité

Un skill est un dossier contenant SKILL.md (frontmatter YAML + corps Markdown), exactement au format des Agent Skills d'Anthropic. Déposez le dossier dans ~/.lucid/skills/<slug>/ et Lucid le prend.

Format

---
name: weekly-report
description: Open Outlook and draft a weekly report email.
version: "1.0"
license: MIT
source: user
---

# Weekly report

1. launch_app: outlook
2. ...

ModèlesSkills

Divulgation progressive

list_skills()

Énumère les skills disponibles.

read_skill(slug)

Récupère le corps complet de SKILL.md.

read_file(path)

Ouvre reference/*.md, scripts/*.py, etc.

Installation en ligne (pilotée par le LLM)

L'exemple d'ouverture du README montre le bootstrap recommandé :

run_shell("git clone https://github.com/anthropics/skills ~/.lucid/skills")
read_file("~/.lucid/skills/pptx/SKILL.md")
learn_tip("PPTX skill installed at ~/.lucid/skills/pptx; entry: build.py")
run_shell("python ~/.lucid/skills/pptx/build.py proposal.docx proposal.pptx")

Le RPC install_skill_url est restreint à l'action utilisateur (le LLM ne peut pas l'appeler) ; le résultat est marqué source: "online" (non fiable) pour que la couche sécurité tranche.

Pourquoi ce design ?

Il existe déjà d'excellents skills dans la nature — le catalogue officiel d'Anthropic, des dépôts communautaires, les bases de connaissances internes des entreprises. Que chaque utilisateur les réinvente est un gâchis et dilue la qualité. On a choisi exprès le format ouvert SKILL.md d'Anthropic : n'importe quel skill écrit ailleurs marche ici tel quel, et la boucle principale devient la découverte — trouver et adopter un bon skill — pas l'écriture. La divulgation progressive (catalogue d'une ligne, read_skill à la demande) fait qu'une centaine de skills ne coûte presque rien en tokens tant que l'agent n'en sollicite aucun.

CHAPITRE 06

Entrée vocale

Push-to-talk · entièrement hors ligne · attentif au focus

Dites une phrase — ou tapez-la — et Lucid ouvre un nouveau fil, interrompt le fil courant ou ajoute au champ d'entrée. Voix et texte tapé partagent le même classifieur d'intention : « annule ça » fonctionne dans les deux cas.

Déclencheur

Maintenir Espace ≥ 5 s

Accessibilité d'abord : pas de combo par défaut. Hook global WH_KEYBOARD_LL ; un repli dans le webview prend le relais quand Lucid lui-même a le focus (sinon Chromium avale la touche).

Configurable

hotkey, hold_threshold_ms, stop_mode (release / tap_again / auto_silence).

Attentif au focus

Si un champ texte ou une composition IME est actif, l'appui long est ignoré — Espace reste Espace.

ASR — local, hors ligne

faster-whisper

Autres moteurs (sherpa-onnx / vosk) conservés pour la compat mais masqués dans l'UI.

Poids locaux

Aucun aller-retour vers le cloud.

≤ 1,5 s

Arrêt → texte sur CPU pour les petits modèles.

Dispatch d'intention EN PROD

Après la transcription — ou pour un texte court (≤ 100 caractères) tapé pendant qu'un fil tourne — un petit LLM de classification (~150 tokens) choisit parmi trois actions :

thread_new
Clairement une nouvelle tâche (« ouvre Notepad et écris hello »).
thread_abort
Stopper la passe en cours (« annule », « stop », « laisse tomber »). Passe par voice_dispatch_abort en fil urgent priority=0 ; le prompt est conscient de la source (typed ou said (voice)) pour ne pas inventer une demande vocale fictive.
dictation_append
Compléter le champ d'entrée (« …et mentionne aussi… »).

Si la confiance est faible, l'overlay propose des chips uniquement quand il retombe en mode result (pas de sink dans la fenêtre principale). L'overlay reste en haut au centre, avec waveform et compte à rebours.

Pourquoi ce design ?

Deux raisons d'avoir intégré la voix dès le départ :

  1. Vitesse pour tout le monde. Parler est environ 3× plus rapide que taper pour la plupart des gens, et on peut le faire en lisant l'écran. La plupart des assistants bureau supposent un clavier uniquement — un plafond inutile.
  2. Accessibilité. Le clavier n'est pas une option pour les utilisateurs avec un handicap moteur, un TMS, ou une blessure temporaire. Un raccourci à maintenir + un ASR entièrement hors ligne signifient que l'entrée fonctionne sans confiance au cloud ni stack d'accessibilité séparé.

Le push-to-talk (au lieu d'un mot-clé toujours à l'écoute) garde le CPU froid, ne fuit jamais d'audio, et évite la « fatigue du mot-clé » qui rend les assistants téléphoniques peu fiables.

CHAPITRE 07

Tests end-to-end

Un JSON · un runner Python · un SKILL.md · Copilot pilote l’itération

Tout le harnais de tests de régression d’un agent GUI tient en trois fichiers : un queries.json avec ~60 tâches réelles, un run.py de 200 lignes qui ne fait qu’enfiler les tâches dans la file prioritaire du sidecar, et un SKILL.md qui apprend à VS Code Copilot comment lire le manifest.json résultant et rédiger le plan de correction de la prochaine itération. Pas de pytest, pas de fixtures, pas de mocks par appli — le bureau lui-même est le banc d’essai.

Entrées — trois fichiers simples

queries.json

~60 tâches réelles dans 15 catégories (Notepad, Calculatrice, Edge, Office, WeChat, Teams…), chacune avec un instruction et un expect_files optionnel. Ajouter de la couverture = un objet JSON.

run.py — un enfileur

Démarre le sidecar, envoie start_task N fois à 300 ms d’intervalle, surveille le mtime de chaque events.jsonl ; si une tâche stagne au-delà de --per-task-cap s, il annule juste celle-là. La file prioritaire du sidecar fait l’ordonnancement.

SKILL.md

Skill Anthropic Agent qui apprend à VS Code Copilot le protocole d’analyse : règles de verdict, fan-out de sous-agents, contrat severity / category_of_fix, langue de sortie. L’analyse est un prompt Copilot — pas du Python.

Boucle d’itération CHAQUE RELEASE

Exécuter
python Test\run.py — le sidecar exécute ~60 requêtes sur le vrai bureau (ne le nettoyez pas avant ; le désordre fait partie du test). Chaque requête réinscrit à la clôture status, final_text et un deliverables_snapshot (fichier existe / taille / mtime / sous-chaîne trouvée, figé à la fermeture) dans manifest.json.
Juger
Verdict en trois couches par requête : run (status == "ok") × goal (un sous-agent Copilot par thread lit instruction + final_text + context.log + 2–3 captures, renvoie pass / partial / fail) × deliverables (snapshot expect_files). Plus de match de sous-chaîne fragile — les paraphrases passent, les succès hallucinés tombent.
Itérer
Fan-out de sous-agents (4–6 en parallèle), chacun finit par un couple machine-lisible severity: / category_of_fix: (prompt / tool / config / new-region / new-app-spec / bug-fix). Copilot les regroupe dans iteration-plan.md : correctifs à fort levier (≥ 2 sous-agents d’accord) d’abord, blockers mono-thread ensuite, tableau complet en dernier. Un baseline en option remonte les régressions.

Effet net : chaque modification de code arrive avec une trace de preuves d’un paragraphe (quelles catégories ont régressé, quel correctif les a débloquées) au lieu d’une impression ; intégrer une nouvelle appli devient « ajouter un objet JSON, relancer, lire le plan ».

Pourquoi ce design ?

Les frameworks classiques de tests GUI (Playwright, pytest+Selenium) obligent le développeur à coder chaque assertion. Or l'assertion, c'est précisément le boulot de l'agent — lui sait à quoi ressemble l'objectif. On a donc inversé : queries.json est le seul contrat, Copilot est le juge. Trois bénéfices : (a) ajouter un cas, c'est une ligne JSON, pas une fixture ; (b) le juge tolère la paraphrase (« un mail de rapport hebdo » ↔ « brouillon d'email »), donc plus de match de chaîne fragile ; (c) le même prompt SKILL.md qui juge la release courante produit aussi le plan d'itération de la suivante. Tests et boucle d'apprentissage deviennent la même boucle.
